# Spec: Project Resources for Harbor Environments

## Problem

Harbor trial pods need three kinds of content:

1. **Runtime** — OS, agent CLI, harness (judge engine, reward bridge)
2. **Project resources** — skills, scripts, .context, CLAUDE.md (the skill under test)
3. **Per-case files** — input.yaml, batch.yaml, hooks, settings.json (the dataset)

Harbor natively handles (1) via `docker_image` and (3) via `environment/` upload.
Project resources (2) have no Harbor equivalent. Current workarounds:

| Approach | Limitation |
|---|---|
| ConfigMap (`AGENT_EVAL_K8S_PROJECT_CONFIGMAP`) | 1 MB etcd limit — fails for large projects (rfe-creator: 79 MB) |
| Baked project image (`FROM base` + COPY) | Couples project to agent; requires rebuild on any change; image in task.toml is per-agent |
| Bind-mount (`AGENT_EVAL_PODMAN_PROJECT_DIR`) | Podman only, not available on remote clusters |

## Design

Use Kubernetes **Image Volumes** (KEP-4639, beta in K8s 1.33+) to mount
project resources as a read-only OCI artifact. This separates project
resources from the runtime image, removes the ConfigMap size limit, and
aligns with Harbor's philosophy (generic image + dynamic agent install).

### Three layers, cleanly separated

```
┌─────────────────────────────────────────────┐
│ Container image (docker_image in task.toml) │  Pre-built, per agent
│   OS + agent CLI + harness                  │  Rarely changes
├─────────────────────────────────────────────┤
│ Image Volume (mounted at /opt/project)      │  Per project, any size
│   skills/ scripts/ .context/ CLAUDE.md      │  Changes with the project
├─────────────────────────────────────────────┤
│ environment/ upload (Harbor runtime)        │  Per case, per trial
│   input.yaml batch.yaml hooks/ .claude/     │  Generated by eval-dataset
└─────────────────────────────────────────────┘
```

### Per-environment behavior

| Environment | Project resources mechanism | Setup |
|---|---|---|
| **Podman** (local) | Bind-mount from host dir | `AGENT_EVAL_PODMAN_PROJECT_DIR=.` |
| **Kubernetes** (generic) | Image Volume from pre-pushed OCI artifact | `AGENT_EVAL_K8S_PROJECT_IMAGE=registry/project:latest` |
| **OpenShift** (`openshift.py`) | Image Volume, built on-cluster via BuildConfig | `AGENT_EVAL_K8S_PROJECT_DIR=/path/to/project` |

### OpenShift on-cluster build

`openshift.py` extends `KubernetesEnvironment` and adds transparent
project image building in `start()`:

1. Check if ImageStream tag exists (cache hit → skip build)
2. If not, create BuildConfig + binary build from the project dir
3. Mount the resulting image as an Image Volume at `/opt/project`
4. Rest of the lifecycle inherits from `KubernetesEnvironment`

The user sets `AGENT_EVAL_K8S_PROJECT_DIR` (same as the current ConfigMap
flow). The environment handles build, push, cache, and mount transparently.
Subsequent runs with unchanged project content skip the build.

### Pod spec (Image Volume)

```yaml
spec:
  containers:
  - name: main
    image: agent-eval-harness-claude:latest
    volumeMounts:
    - name: project
      mountPath: /opt/project
      readOnly: true
  volumes:
  - name: project
    image:
      reference: image-registry.openshift-image-registry.svc:5000/ns/project:latest
      pullPolicy: IfNotPresent
```

### Workspace symlinks

The Image Volume mounts read-only at `/opt/project`. The agent needs
skills discoverable at `~/.claude/skills/` and project files in the
workspace. Symlinks bridge the gap:

```
/workspace/.claude/skills/ → /opt/project/.claude/skills/  (read-only)
/workspace/CLAUDE.md       → /opt/project/CLAUDE.md        (read-only)
/workspace/scripts/        → /opt/project/scripts/         (read-only)
/workspace/.context/       → /opt/project/.context/        (read-only)
```

These are created by the pod entrypoint (the log-tailing loop in
`_pod_manifest()`) before entering the tail loop, or by a setup exec
in `start()` after the pod is running.

`/workspace/.claude/` must be a real writable directory (not a symlink)
because Harbor uploads `environment/.claude/settings.json` at runtime.
Only `skills/` inside it is a symlink to the read-only mount.

### OCI artifact format

The project image is a minimal OCI image containing only text files:

```
.claude/skills/rfe.speedrun/SKILL.md
.claude/skills/rfe.create/SKILL.md
scripts/frontmatter.py
scripts/state.py
.context/architecture-context/...
CLAUDE.md
```

Built with `oras push`, `buildah`, or OpenShift BuildConfig. No
Dockerfile, no runtime — just content files with a filesystem layout.

## What doesn't change

- **eval-dataset** — generates test cases and task packages. References
  the runtime image in task.toml via `--image`. Does not build or push
  project images.
- **eval-run** — runs task packages. Does not know about project images.
  The environment handles mounting.
- **task.toml** — `docker_image` stays the runtime image (generic or
  per-agent). No project image reference in the task spec.
- **Podman** — bind-mount from host, no OCI artifact needed.
- **Harbor upstream** — no changes required. Image Volumes are a K8s
  feature, transparent to Harbor.

## Migration path

1. **Phase 1**: Add Image Volume support to `kubernetes.py` alongside
   the existing ConfigMap path. `AGENT_EVAL_K8S_PROJECT_IMAGE` → Image
   Volume. `AGENT_EVAL_K8S_PROJECT_CONFIGMAP` → ConfigMap (kept for
   small projects / older K8s).

2. **Phase 2**: Create `openshift.py` extending `KubernetesEnvironment`
   with BuildConfig-based project image building from
   `AGENT_EVAL_K8S_PROJECT_DIR`.

3. **Phase 3**: Deprecate ConfigMap path once Image Volumes are proven
   stable. Remove project-specific Containerfiles from project repos
   (rfe-creator's `deploy/Containerfile` becomes unnecessary).

## Directory layout considerations

- Image Volume is always read-only — scripts that write to `/opt/project`
  will fail. Project resources should be treated as immutable.
- Symlinks across mount boundaries work on Linux (pod filesystem).
- The OCI artifact build tool must preserve the directory tree (not flatten).
- BuildConfig creates an ImageStream in the namespace — needs cleanup
  on teardown (same `managed-by: agent-eval-harness` label pattern).
